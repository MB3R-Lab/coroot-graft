# coroot-graft Architecture

## Design Boundary

`coroot-graft` is not a third core engine.

- Discovery stays in upstream `Bering`.
- Resilience analysis and gate stay in upstream `Sheaft`.
- `coroot-graft` only adds Coroot-specific extraction, orchestration, export, and operational packaging.

## Data Flow

1. `coroot-graft` authenticates against Coroot using `/api/login`.
2. It reads Coroot project context from:
   - `overview/map`
   - `app/{app}`
   - optionally `app/{app}/tracing`
3. It converts the Coroot snapshot into explicit Bering `topology_api` input.
4. It runs `bering discover`.
5. It runs `sheaft run` on the produced Bering snapshot artifact.
6. It stores run artifacts under `.coroot-graft/projects/{project}/`.
7. It exports latest results as Prometheus metrics and optionally installs a managed Coroot dashboard.

## Snapshot Layers

There are three different "snapshot" concepts in this project and they should not be conflated.

### 1. Coroot snapshot

This is the transient in-memory structure assembled by `coroot-graft` during a sync.

It contains:

- applications
- dependencies
- replica counts
- optional traced endpoints

It is collected from Coroot HTTP APIs on every startup, interval sync, webhook-triggered sync, or manual `sync` run.

### 2. Bering `topology_api` input

This is the explicit topology document generated by `coroot-graft` from the Coroot snapshot.

It contains:

- `services`
- `edges`
- `endpoints`
- `source`

This is the format handed to `bering discover`.

### 3. Bering snapshot artifact

This is upstream `Bering`'s emitted `io.mb3r.bering.snapshot@1.3.0` artifact.

It is the canonical MB3R discovery artifact later consumed by `Sheaft`.

## Posture Versus Health

`coroot-graft` publishes resilience posture, not live health status.

That distinction matters.

Coroot built-in views answer:

- what is unhealthy right now?
- which service is down?
- which alerts or incidents are active?

`coroot-graft` answers:

- how resilient is the currently observed topology under configured failure modes?
- which entrypoints are most fragile?
- does this topology pass the resilience gate?

In other words:

- Coroot health is observed runtime state.
- Sheaft posture is simulated resilience over the current discovered model.

Because of that, posture is recomputed every sync, but it only changes when the discovered model or analysis inputs change.

Stopping a runtime service does not automatically change posture if Coroot still exposes the same topology membership and dependency graph in the source snapshot.

## Service IDs

Service IDs are the raw Coroot `ApplicationId` strings.

Example:

```text
cluster-a:default:Deployment:frontend
```

Why:

- they are stable and unique across namespaces and clusters
- they do not require inventing a second naming scheme inside `coroot-graft`
- they map cleanly back to Coroot application objects

## Endpoint Strategy

Default endpoint mode is `service`.

That means:

- each Coroot application gets one synthetic entry endpoint
- Sheaft then derives journeys from the blocking sync graph starting at that entry service

Optional endpoint mode is `trace_http`.

That mode:

- samples Coroot tracing data
- extracts `http.method` plus `http.route` / `url.path` / `http.target`
- falls back to synthetic service endpoints when no usable HTTP spans are available

This keeps the MVP deterministic by default while allowing higher-fidelity entrypoints when Coroot tracing data is present.

## Edge Semantics

Default edge normalization:

- `kind=sync`
- `blocking=true`

Async inference is intentionally narrow:

- queue-like dependencies (`kafka`, `rabbitmq`, `nats`) are mapped to `kind=async`, `blocking=false`
- explicit `edge_overrides` in config win over heuristics

This keeps `coroot-graft` as a normalization layer, not a second discovery engine.

## Coroot Integration Surface

Results are published back to Coroot through documented or observable external surfaces:

- `/metrics` for Coroot custom metrics scraping
- `/webhooks/coroot/{project}` for event-driven reruns from Coroot alert/deployment/incident webhooks
- `/api/project/{project}/dashboards` for managed dashboard installation

## Known MVP Limits

- Coroot topology extraction uses authenticated HTTP APIs rather than a formal extension SDK because Coroot does not expose a dedicated extension SDK in `main.go`.
- Dashboard installation relies on the observed Coroot dashboard API contract from the server source, not on a public dashboard-import API.
- When `trace_http` is disabled or Coroot tracing is unavailable, endpoint coverage is service-level rather than per-route.
- In the current MVP, topology membership is primarily driven by Coroot application inventory rather than a trace-only rolling discovery window.
- As a result, `coroot-graft` should be read as "live recomputed posture from current Coroot-derived topology", not as a direct live health score.
